# LockPro 改造需求与方案（2026-04-12）

## 1. 背景
本轮目标是把交易中心从“用户手动从订单簿下单”为主，调整为“后台可按用户批量生成订单”为主；前端订单簿入口临时隐藏但保留代码，后续可随时放开。

## 2. 本轮范围（已确认）
1. 前端隐藏“订单簿”区域（仅注释隐藏，不做删除）。
2. 原订单簿占位由“收益曲线”补满整行展示。
3. 前端订单记录表头“提交时间”改为“订单时间”。
4. 后台隐藏“订单管理”模块（仅注释隐藏菜单与页面入口，保留代码）。
5. 后台用户管理新增“本金”列（展示该用户本金总额）。
6. 后台用户管理支持“自动生成订单”。
7. 自动订单支持：
   - 输入总金额（<= 可用本金）
   - 输入订单条数
   - 订单号按现有规则生成
   - 冻结时间区间（默认配置 + 本次覆盖）
   - 收益区间（默认配置 + 本次覆盖）
   - 项目名随机（允许重复，跨用户允许重复）
   - 订单时间区间（精确到 年/月/日/时/分）
8. 自动生成订单必须同步体现在该用户前端“订单记录”。

## 3. 关键业务口径
### 3.1 订单时间与结算
- 订单时间在配置区间内随机生成。
- `settle_at = order_time + 冻结时长(随机)`。

### 3.2 立即结算规则（已确认）
- 若某条生成订单 `settle_at <= 当前时间`：
  - 该订单生成时直接写为“已完成”。
  - 收益在同一事务中立即入账。
  - 不等待下一次结算轮询。
- 若 `settle_at > 当前时间`：
  - 该订单写为“进行中”。
  - 到期后按既有结算机制完成收益入账。

### 3.3 本金口径
- 自动生成订单仅允许使用用户可用本金（`totalAmount <= principal_available`）。
- 管理员“加余额”仍视为增加充值本金。

## 4. 后端设计（目标形态）
## 4.1 数据表变更
1. `users`
- 复用现有：
  - `principal_available`
  - `frozen_balance`
  - `profit_available`
  - `total_profit`
- 用户管理“本金”列展示建议口径：
  - `principal_total = principal_available + frozen_balance`

2. `order_generation_defaults`（新增）
- 用于后台“默认配置”：
  - `freeze_min`, `freeze_max`
  - `rate_min`, `rate_max`
  - `time_start`, `time_end`（到分钟）
  - `project_pool`（JSON）
  - `updated_at`

3. `user_orders`
- 复用现有字段：
  - `submitted_at` 作为“订单时间”展示。
  - `settle_at` 作为结算时间。
  - `status`：`running` / `done`。

## 4.2 接口设计（建议）
1. `GET /api/admin/order-generation-defaults`
- 获取自动订单默认配置。

2. `PUT /api/admin/order-generation-defaults`
- 更新自动订单默认配置。

3. `POST /api/admin/users/:id/generate-orders`
- 批量生成订单。
- 入参：
  - `totalAmount`
  - `orderCount`
  - 可选覆盖：`freezeMin/freezeMax`、`rateMin/rateMax`、`timeStart/timeEnd`、`projectPool`
- 逻辑：
  - 校验本金与参数合法性
  - 按条数拆分总金额
  - 逐条生成订单号、项目名、订单时间、冻结时长
  - 计算 `settle_at`
  - `settle_at <= now` 的订单立即完成并入账
  - 其余订单置为进行中并冻结本金

## 5. 前端/后台 UI 改造点
## 5.1 前端（`frontend`）
1. 交易中心：
- 注释隐藏订单簿卡片（保留代码）。
- 曲线图卡片改为整行占满。

2. 订单记录：
- 列名“提交时间”->“订单时间”。

## 5.2 后台（`admin`）
1. 侧边栏：
- 注释隐藏“订单管理”按钮（保留代码）。

2. 用户管理：
- 列表新增“本金”列。
- 操作区新增“生成订单”按钮（弹窗）。

3. 默认配置：
- 在用户管理页面新增“自动订单默认配置”卡片。

## 6. 项目名随机策略（尽可能多）
采用双层策略：
1. 优先使用后台可维护项目池（JSON/文本列表）。
2. 当项目池不足或需要扩展时，使用组合生成器（前缀 + 核心词 + 后缀码），理论可持续生成大量名称。

说明：项目名允许重复，不要求全局唯一。

## 7. 事务与一致性要求
批量生成订单必须在单事务内完成：
1. 插入 `user_orders` 多条记录。
2. 同步更新用户资金字段。
3. 任一环节失败则整体回滚。

## 8. 风险与注意事项
1. 使用历史时间批量生成时，会出现大量“已完成”订单与即时收益入账，属于预期行为。
2. 时间区间建议校验：`start <= end`，并限制跨度防止一次生成超大历史数据。
3. 总金额拆分需做“分”级精度处理，保证各条金额之和严格等于输入总金额。

## 9. 联调验收清单（建议）
1. 默认配置保存/读取是否一致。
2. 生成订单（全未来时间）应全部为“进行中”。
3. 生成订单（全历史时间）应全部“已完成”且收益即时入账。
4. 混合时间区间应同时出现进行中/已完成两类。
5. 前端订单记录是否立即可见后台生成结果。
6. 前端“订单时间”展示是否正确。
7. 隐藏模块是否仅隐藏入口，代码仍可恢复。
